## 🟡 Important

### I1 🟡 Port fallback can split durable OAuth state

**Where**
`src/auth/provider.ts:52`

**Code**
```ts
return undefined;
```

**Why**
`src/mcp/client.ts:26` — `const savedPort = extractPortFromClientInfo(tokenStore.readClientInfo());` passes a persisted redirect port into the provider via `src/mcp/client.ts:28` — `preferredPort: savedPort,`.
When `clientInformation()` returns `undefined` on fallback port, the SDK can write a new fallback-port `client.json` before token exchange succeeds, while the old `tokens.json` remains preserved as shown by `src/auth/provider.test.ts:120` — `expect(store.readTokens()).toEqual(tokens);`.
That recreates the split OAuth-state class this change is trying to avoid.

**Fix**
Make fallback re-registration atomic with token replacement: stage the new client registration until token save succeeds, or clear/rollback the coupled MCP OAuth files when browser auth on a fallback port actually starts.

### I2 🟡 UnauthorizedError recovery assumes callback initialization ordering

**Where**
`src/mcp/client.ts:46`

**Code**
```ts
const code = await provider.waitForCallback();
```

**Why**
The provider initializes the callback promise only in `src/auth/provider.ts:89` — `this.callbackPromise ??= this.callbackServer.waitForCallback();`.
If `UnauthorizedError` reaches `MCPConnection` before the SDK has called `redirectToAuthorization()`, `src/auth/provider.ts:97` — `"OAuth callback not started",` becomes the recovery path instead of browser auth.

**Fix**
Make `waitForCallback()` defensively start/wait when lazy callback is enabled, or have `MCPConnection` explicitly start the provider callback before awaiting it; add a regression test for the `UnauthorizedError` path.
